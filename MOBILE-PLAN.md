# خطة بناء تطبيق «القرآن الكريم» للموبايل (Android + iOS) بـ Flutter

> نقل تطبيق سطح المكتب الحالي (WPF / .NET 9) إلى تطبيق موبايل بنفس المزايا، باستخدام **Flutter**.
> تطوير: **شركة الرائد للحلول البرمجية**.

## القرارات المعتمدة
- **البيانات:** نشحن `quran.db` المبنية جاهزة كـ asset وننسخها أول تشغيل (بدون إعادة فهرسة على الجهاز).
- **الصوت:** تحميل مسبق ثم تشغيل أوفلاين (نفس سلوك النسخة الحالية).
- **إدارة الحالة:** Riverpod.

## ليش Flutter؟
- كود واحد لـ Android و iOS (وإمكانية نسخة ويب/ويندوز لاحقاً).
- دعم ممتاز للعربية و RTL والخطوط المخصّصة.
- SQLite محلي عبر `sqflite` — نفس فلسفة البيانات المحلية الحالية.

## الفلسفة: نعيد استخدام البيانات لا الكود
كود C#/XAML ما ينتقل لـ Dart، لكن البيانات والمنطق المُجهّز ينتقل:

| من المشروع الحالي | بالموبايل |
|---|---|
| `quran.db` الجاهزة | تُشحن كـ asset، تُنسخ أول تشغيل، تُفتح للقراءة فقط |
| منطق `ArabicText.Normalize` | يُعاد كتابته بـ Dart حرفياً (حذف Mn/Cf، توحيد أإآٱ→ا، ىئي→ي، ؤ→و، ة→ه) |
| منطق `Grammar.Describe` (الإعراب) | يُعاد كتابته بـ Dart فوق نفس جداول الصرف |
| روابط `everyayah.com` | نفسها، التحميل عبر `dio` والتخزين عبر `path_provider` |

## الحزم الأساسية
`flutter_riverpod`، `sqflite`، `go_router`، `just_audio` + `audio_session`، `dio`، `shared_preferences`، `path_provider`.

## البنية المقترحة
```
lib/
  core/
    arabic_text.dart        // Normalize — منقول من ArabicText
    grammar.dart            // الإعراب — منقول من Grammar.Describe
    constants.dart
  data/
    db/quran_database.dart   // فتح sqflite + نسخ الـ asset أول مرة
    repositories/            // AyahRepository, SearchRepository, TafsirRepository
    models/                  // Ayah, SearchResult, Reciter, ...
  features/
    mushaf/                  // عرض الصفحة بتبرير متّصل
    search/                  // البحث النصّي + بالجذر + النتائج
    ayah_detail/             // الإعراب + التفسير + التلاوة
    settings/                // الخيارات
  services/
    audio_service.dart       // just_audio + تحميل مسبق
    settings_service.dart    // shared_preferences
  app.dart / main.dart
```

## المراحل (Milestones)

### المرحلة 0 — التحضير (نصف يوم)
- إنشاء مشروع Flutter، RTL افتراضي (locale عربي + `Directionality`)، تضمين خطوط المصحف كـ assets.
- نسخ `quran.db` الجاهزة إلى `assets/db/`.

### المرحلة 1 — طبقة البيانات (1–2 يوم)
- فتح الـ DB ونسخها أول تشغيل من الـ asset لمجلد التطبيق.
- نقل `ArabicText.Normalize` لـ Dart + اختبارات وحدة تطابق نتائج C#.
- Repositories: آية/صفحة/سورة، التفسير، الصرف.

### المرحلة 2 — عرض المصحف (2–3 يوم)
- تبرير متّصل: `Text.rich` مع `TextAlign.justify` و `WidgetSpan` للكلمات القابلة للنقر.
- تباعد السطور عبر `TextStyle.height` (نسبة، أسهل من WPF).
- تقليب الصفحات بسحب أفقي (`PageView`).
- صفحة واحدة (افتراضي) + شريط معلومات (جزء/حزب/سجدة/إحصائيات).

### المرحلة 3 — البحث (2 يوم)
- بحث نصّي + بالجذر، نفس الأعمدة (`NormText/LightText/NormUthmani/LightUthmani`, `Root/NormLemma/NormForm`).
- صفحة نتائج، النقر يفتح صفحة المصحف.
- شريط إحصائيات.

### المرحلة 4 — تفاصيل الآية (2–3 يوم)
- نقر على كلمة → الإعراب (نقل `Grammar`).
- التفسير الميسَّر أوفلاين من الـ DB.
- التلاوة: 4 قُرّاء (الحصري/المنشاوي/عبد الباسط/السديس)، تحميل مسبق ثم أوفلاين، تمييز الآية الجارية، النقر ينقل الصوت — عبر `just_audio` + تخزين في `path_provider`.

### المرحلة 5 — الإعدادات (1 يوم)
- خيارات البحث (FoldLetters/BothRasm)، شكل النسخ، الخط/الحجم، إدارة تحميل الصوت (حالة كل قارئ/تحميل/حذف).

### المرحلة 6 — التلميع والنشر (2–3 يوم)
- ثيم فاتح/داكن، أيقونة، شاشة بداية.
- بناء `aab`/`apk` للأندرويد و `ipa` لـ iOS، الرفع على المتاجر.

**التقدير الإجمالي: ~12–17 يوم عمل.**

## تنبيهات تقنية للنقل
- مكافئ WPF `LineHeight` (بكسل) هو `TextStyle.height` بـ Flutter (نسبة) — أبسط.
- لازم تطابق نتائج `ArabicText.Normalize` بين C# و Dart حرف-بحرف، وإلا يفسد البحث؛ نضيف اختبارات وحدة بعيّنات.
- الـ `quran.db` المشحونة لازم تكون مبنية بأحدث منطق تطبيع (نفس النسخة المنشورة على ويندوز).
- خطوط المصحف لازم تُرخَّص للتضمين بالموبايل.
